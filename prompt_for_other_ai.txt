## 1. 依頼の概要
現在、私が管理している「Typstのテンプレートリポジトリ」について、今後のアップデートに向けた大規模な改修（インストーラーの作成、テンプレート展開方法の整理、スクリプトの強化など）を計画しています。

この改修を効率的かつ堅牢に進めるため、あなたと設計の壁打ち（ブレインストーミング、技術選定、設計書のブラッシュアップ）を行いたいです。

## 2. 現在のリポジトリ構成と各ファイルの役割

現在のリポジトリのディレクトリ構成と、それぞれのファイルの役割は以下の通りです。

```text
Typst_Template/
├── init_module/               # init.ps1によって親リポジトリにコピーされる設定ファイル群
│   ├── .github/
│   │   └── workflows/
│   │       └── compile-typst.yml  # Typstコンパイルを自動化するGitHub Actions
│   ├── .vscode/
│   │   ├── settings.json          # VS Code用のTypst設定（[typst] quickSuggestions 等）
│   │   └── typst.code-snippets    # VS Code用のTypstスニペット
│   └── .gitignore             # 親リポジトリ側のPDF出力を除外する設定 (.pdf)
├── demo/
│   ├── demo_slide.typ         # スライドテンプレートのデモ用ファイル
│   └── demo_slide.pdf
├── examples/                  # テンプレートの使用例
│   ├── ex-refs.bib            # 参考文献用BibTeX
│   ├── report_ja.typ          # 日本語レポート作成例
│   ├── thesis_en.typ          # 英語論文作成例
│   └── thesis_ja.typ          # 日本語論文作成例
├── test/
│   └── test_slide.typ         # 動作テスト用ファイル
├── init.ps1                   # 親リポジトリ（Submodule先）で実行し、init_module の設定をコピー・マージするPowerShellスクリプト
├── update_packages.ps1        # .typファイル内で使用している外部パッケージ（@preview/...）のバージョンを最新版に自動更新するPowerShellスクリプト
├── lib.typ                    # 初代テンプレート（主に日本語レポート用、一部論文対応）
├── lib_v2.typ                 # 改良版テンプレート（多言語対応、カバーページ/抄録/目次/章番号調整のコンポーネント化）
├── slide.typ                  # スライド用テンプレート（Touying, cetz, fletcher, theorionパッケージ等の設定を含む）
├── utils.typ                  # 共通ユーティリティ（数式記法マクロ、解答欄・囲み枠・単位出力などの定義）
├── plan.md                    # 今後の開発計画（今回の改修のベースとなる文書）
└── README.md                  # リポジトリの取扱説明書
```

### 各主要ファイルの機能概要
- **`lib_v2.typ`**: 論文やレポート用のコアテンプレートです。日本語(`ja`)と英語(`en`)の切り替え、表紙（カバーページ）生成、概要（Abstract）ブロック、ローマ数字からアラビア数字にリセットされる目次（TOC）などを備えています。
- **`slide.typ`**: スライド作成用のテンプレートで、`touying` をベースに `cetz` (描画) や `fletcher` (ダイアグラム)、`theorion` (定理環境) などの外部パッケージを統合し、日本語フォント（Harano Aji Gothic等）の設定を行っています。
- **`utils.typ`**: 数学の順列・組み合わせ (`combination`, `permutation`, `hcombination`)、解答用ボックス (`ans`)、枠囲み (`crect`, `frect`)、物理単位用フォーマッタ (`u`) などのカスタムマクロを提供します。
- **`init.ps1`**: ユーザーがこのテンプレートをGit submoduleとして自分のリポジトリに追加した際、親リポジトリのルートで実行することで、VS Code用の設定・スニペットやGitHub Actionsのワークフローを自動的に配備・マージします。

## 3. 今後の開発計画（`plan.md` の内容）

リポジトリ内に作成した `plan.md` の内容は以下の通りです。

```markdown
# 今後の開発案
## init.ps1への機能追加
もし実行時のPCでtypstがない場合には，wingetからインストールさせる．

## github releaseを利用した，install_typst-template.exe(仮)でのテンプレートインストール
githubのReleaseから.exeファイルをダウンロードし，ダブルクリックで実行するだけで，このリポジトリ内のテンプレートをパスを指定して置く．

パスは，C:\Users\%USERNAME%\typst\templates\Typst_Template\(仮)に置かれるようにする．

この変更に合わせ，スニペットによるインポート部分を絶対パスにすれば，どこからでもテンプレートを使用可能．

.exeファイル実行時には，init.ps1も同時に実行（typst本体のwingetインストールなど）

## submoduleと，.exeによるインストールの2通りに区別
### submodule
今までと同様

### .exeによるインストール
init.ps1による実行では，指定したフォルダへの.vscode,.github等の各種ファイルが置かれない．

よって，init.ps1とは別の方法でtypst本体のインストールなどを行う

#### .exeの仕様
- テンプレートインストール場所を指定（デフォルトはC:\Users\%USERNAME%\typst\templates\Typst_Template）
- wingetを利用したtypst本体のインストール（必要に応じて）
- .vscode,.github等の，vscodeやgithubでの管理専用ファイルはインストールさせない．（必要な場合はsubmoduleによる実装）
- Harano Ajiフォントのダウンロード及びインストールも，.exe実行時にできればなお良い
- インストール先を指定するUIを導入（ボタンでフォルダ選択など）（仮）
この仕様変更に応じたsubmoduleの仕様変更も必要かもしれない．

submodule時には完全に必要なファイルのみを持ってくるようにする．（.vscodeや.githubなど）

submodule時にテンプレートファイル等も持ってくるかどうかは要検討

## README.mdの内容を修正
submodule及び.exeについての説明を追加
```

## 4. ディスカッションしたい論点・質問

上記の構成と開発計画を踏まえ、以下の点についてアドバイスや設計の提案をいただけますでしょうか。

### 論点 1: `.exe` インストーラーの技術選定と実装方法
- Windows向けにGUI付き（またはシンプルなCUI）の `.exe` インストーラーを開発する場合、どのプログラミング言語・ツールチェーン（例: Go + walk/fyne, Rust + egui/slint, Python + PyInstaller, C#/.NET, あるいは PowerShell をパッケージ化するもの）が最適でしょうか？
- ユーザーにランタイムのインストールを要求せず、単一バイナリで動作し、サイズが軽量であるものが望ましいです。

### 論点 2: ローカルテンプレートの配置とインポート方法の設計
- Typstはローカルのディレクトリ（`C:\Users\%USERNAME%\AppData\Local\typst\packages\local\<package>\<version>` 等）にパッケージを配置することで、`#import "@local/mypackage:0.1.0": *` のようにインポートできる仕組み（ローカルパッケージ）があります。
- `plan.md` にある `C:\Users\%USERNAME%\typst\templates\Typst_Template` への配置と、Typst公式のローカルパッケージ仕様（`AppData\Local\typst\packages`）のどちらを利用するのが、今後の汎用性や使いやすさの観点から望ましいでしょうか？

### 論点 3: Harano Aji フォントの自動インストール
- `Harano Aji Mincho` / `Harano Aji Gothic` フォントは日本語のTypstコンパイルに不可欠ですが、インストーラー内でこれをダウンロードし、Windowsシステムに自動インストールする（ユーザーフォルダフォント、またはシステムフォントとして登録する）ための最も安定した手法は何でしょうか？

### 論点 4: Submodule方式とインストーラー方式の棲み分け・ファイル構成の最適化
- サブモジュールとしてこのリポジトリを取り込む場合、親リポジトリに `.vscode` や `.github` をコピーしたいという需要があります。
- 一方、インストーラー経由の場合、純粋にテンプレート（`lib.typ` 等）だけをローカルパッケージとして登録すればよく、`.vscode` 等のメタファイルは不要です。
- この2つのユースケースを共存させるために、現在のディレクトリ構成（`init_module` の配置など）を変更・整理する必要はありますか？

### 論点 5: その他、テンプレートのアップデートに向けた改善提案
- `lib_v2.typ` や `slide.typ` の構成、あるいはパッケージ自動更新スクリプト（`update_packages.ps1`）に関して、さらに堅牢にするためのアイデアがあれば教えてください。
